# pfSense เชื่อมต่อ OpenVPN แล้วหลุดบ่อย

เพื่อให้เข้าใจปัญหาและวิธีแก้ไขอย่างลึกซึ้ง สาเหตุที่ pfSense เชื่อมต่อ OpenVPN แล้วหลุดบ่อย (Frequent Disconnects) สามารถแบ่งออกเป็น 5 ด้านหลัก พร้อมรายละเอียดเชิงเทคนิคและวิธีจัดการดังนี้
------------------------------
## 1. ปัญหาด้านเครือข่ายและการรับส่งข้อมูล (Network & Transport Layer)

* Packet Loss และ Connection Timeout: OpenVPN ตรวจสอบการเชื่อมต่อผ่านฟังก์ชัน keepalive (การส่งสัญญาณ Ping ตรวจสอบกันระหว่างฝั่ง Client และ Server) หากอินเทอร์เน็ตฝั่งใดฝั่งหนึ่งแกว่ง หรือเกิด Packet Loss สูง สัญญาณ Ping นี้จะหายไป เมื่อครบกำหนดเวลา ระบบจะถือว่าสายหลุดและเริ่มตัดการเชื่อมต่อเพื่อต่อใหม่
* การบล็อกพอร์ตจาก ISP หรือ Firewall ระหว่างทาง: หากเลือกใช้โปรโตคอล UDP แม้จะทำงานได้เร็ว แต่อินเทอร์เน็ตบางประเภท (เช่น เน็ตหอพัก เน็ตองค์กร หรือเน็ตมือถือบางค่าย) อาจจำกัดแบนด์วิดท์ของ UDP หรือสุ่มตัดการเชื่อมต่อเมื่อเห็นว่ามีการรับส่งข้อมูลผ่านพอร์ตเดิมเป็นเวลานาน
* วิธีแก้เชิงลึก:
* ลองเปลี่ยนโปรโตคอลจาก UDP ไปเป็น TCP (พอร์ต 443) แม้จะช้ากว่าเพราะต้องรอการยืนยันข้อมูล แต่พอร์ต 443 (HTTPS) มักจะไม่ถูกบล็อกและมีความเสถียรกับเครือข่ายที่ไม่นิ่งมากกว่า
   * ตั้งค่า keepalive ให้ยืดหยุ่นขึ้นในช่อง Custom Options เช่น keepalive 10 120 (ส่ง Ping ทุก 10 วินาที แต่ให้รอถึง 120 วินาทีก่อนจะตัดการเชื่อมต่อ)

------------------------------
## 2. ข้อจำกัดของฮาร์ดแวร์และการเข้ารหัส (Hardware & Cryptography)

* Single-Thread Bottleneck: สถาปัตยกรรมของ OpenVPN ถูกออกแบบมาให้ทำงานแบบ Single-threaded (ใช้งาน CPU เพียงคอร์เดียวในการประมวลผลต่อหนึ่งอุโมงค์เชื่อมต่อ) หาก pfSense รันอยู่บนซีพียูรุ่นเก่า ซีพียูตระกูลประหยัดพลังงาน (เช่น Intel Celeron รุ่นเก่า, Atom) หรือบอร์ดเราเตอร์ขนาดเล็ก (Mini PC) เมื่อมีการรับส่งข้อมูลปริมาณมาก ซีพียูคอร์นั้นจะทำงานเต็ม 100% จนเกิดอาการคอขวด ประมวลผลแพ็กเก็ตไม่ทัน และทำให้ VPN หลุด
* Thermal Throttling (ปัญหาความร้อน): อุปกรณ์ Mini PC หลายรุ่นไม่มีพัดลม (Passive Cooling) เมื่อซีพียูถูกเร่งประสิทธิภาพเพื่อเข้ารหัสข้อมูล VPN เป็นเวลานาน ความร้อนจะสะสมสูงเกินขีดจำกัด ตัวซีพียูจะลดความเร็วตัวเองลงอย่างกะทันหัน ส่งผลให้การเชื่อมต่อ VPN ขาดหาย
* วิธีแก้เชิงลึก:
* เปิดใช้งาน Hardware Crypto ในหน้า System > Advanced > Miscellaneous ตรงหัวข้อ Cryptographic Hardware ให้เลือกเป็น AES-NI (หากซีพียูรองรับ) เพื่อยกหน้าที่การเข้ารหัสไปให้ชิปเซ็ตโดยตรง ลดภาระของซีพียูหลัก
   * เปลี่ยนอัลกอริทึมการเข้ารหัส (Data Encryption) ในหน้าตั้งค่า OpenVPN ให้เป็น AES-128-GCM หรือ CHACHA20-POLY1305 ซึ่งประมวลผลได้เร็วกว่าและกินทรัพยากรน้อยกว่า AES-256-CBC รุ่นเก่า
   * หากฮาร์ดแวร์ไม่ไหวจริงๆ แนะนำให้พิจารณาเปลี่ยนไปใช้ WireGuard แทน OpenVPN เนื่องจาก WireGuard รองรับการทำงานแบบ Multi-threaded (ใช้ CPU ได้ทุกคอร์) และกินทรัพยากรน้อยกว่ามาก

## 3. ปัญหาขนาดของแพ็กเก็ตข้อมูล (MTU / MSS Clipping)

* สาเหตุ: ข้อมูลอินเทอร์เน็ตปกติมีขนาดมาตรฐาน (MTU) อยู่ที่ 1500 bytes แต่เมื่อข้อมูลถูกส่งผ่าน OpenVPN ระบบจะต้องห่อหุ้มข้อมูลนั้นด้วย "หัวข้อและรหัสความปลอดภัย" (VPN Overhead) เพิ่มเข้าไปอีกประมาณ 60-80 bytes ทำให้ขนาดแพ็กเก็ตเกิน 1500 bytes
เมื่อแพ็กเก็ตใหญ่เกินไป อุปกรณ์เราเตอร์ระหว่างทางที่ไม่รองรับการย่อยแพ็กเก็ต (Packet Fragmentation) จะทำการทิ้ง (Drop) แพ็กเก็ตนั้นทันที ส่งผลให้หน้าเว็บโหลดค้าง และ VPN ตัดการเชื่อมต่อเนื่องจากข้อมูลขาดหาย
* วิธีแก้เชิงลึก:
* บังคับลดขนาดแพ็กเก็ตเพื่อเผื่อพื้นที่ให้ตัวหุ้ม VPN โดยใส่คำสั่งในช่อง Custom Options ของ OpenVPN:
   * tun-mtu 1400 (จำกัดขนาดอุโมงค์ส่งข้อมูลไว้ที่ 1400 bytes)
      * mssfix 1360 (บังคับให้ข้อมูล TCP ปรับขนาดเล็กลงตามความเหมาะสม)
   
------------------------------
## 4. การชนกันของ IP และการตั้งค่า Config ซ้ำซ้อน (Routing & Authentication)

* Subnet ชนกัน (IP Conflict): หากวง Network ภายในบ้านของคุณ (เช่น 192.168.1.X) ไปตรงกับวง Network ของปลายทางที่ VPN วิ่งไปหา ระบบจะเกิดอาการสับสนในการค้นหาเส้นทาง (Routing Loop) ทำให้เน็ตเวิร์กค้างและหลุด
* การใช้บัญชี/ใบรับรองซ้ำกัน: ในกรณีที่ pfSense ทำหน้าที่เป็น OpenVPN Server หากคุณสร้าง User เกรดเดียว หรือใช้ไฟล์ Certification เดียวกัน แจกจ่ายให้พนักงานหรืออุปกรณ์หลายเครื่องนำไปล็อกอินพร้อมกัน ตัว OpenVPN Server จะมองว่าเป็นการล็อกอินซ้อน และจะทำลายการเชื่อมต่อ (Kick) ของอุปกรณ์เครื่องเก่าทิ้งทันทีเมื่อเครื่องใหม่ต่อเข้ามา
* วิธีแก้เชิงลึก:
* เปลี่ยนวง LAN ภายในบ้านให้เป็นหมายเลขที่แปลกตา เช่น 10.50.10.X หรือ 192.168.89.X เพื่อไม่ให้ไปชนกับเน็ตเวิร์กปลายทาง
   * หากจำเป็นต้องใช้ User เดียวกันล็อกอินหลายอุปกรณ์จริงๆ ให้ไปที่หน้าตั้งค่า OpenVPN Server มองหาหัวข้อ Duplicate Connections แล้วติ๊กถูกที่ Allow multiple connections from the same user (อย่างไรก็ตาม วิธีที่ดีที่สุดคือการสร้างแยก 1 User ต่อ 1 อุปกรณ์เพื่อความปลอดภัย)

------------------------------
## 5. วิธีการตรวจสอบผ่าน Log (Root Cause Analysis)
เพื่อที่จะไม่ต้องเดาสุ่ม คุณสามารถดูสาเหตุที่แท้จริงได้โดยตรงจาก pfSense:

   1. ไปที่เมนู Status > System Logs > OpenVPN
   2. มองหาข้อความแจ้งเตือนสีแดงหรือโค้ดผิดพลาด (Error Code) เช่น:
   * Inactivity timeout... restarting: เกิดจากอินเทอร์เน็ตขาดหาย หรือฝั่งตรงข้ามไม่ตอบสนอง (โยงไปที่ปัญหาข้อ 1 หรือ 3)
      * AUTH_FAILED: ชื่อผู้ใช้ รหัสผ่าน หรือ TLS Key ไม่ถูกต้องหรือหมดอายุ
      * TLS Error: TLS handshake failed: บล็อกพอร์ต, ตั้งเวลาเครื่อง (Time/NTP) ไม่ตรงกัน หรือไฟล์ Certificate ไม่ตรงกับฝั่ง Server
   
หากต้องการให้ช่วยวิเคราะห์ Log สามารถคัดลอกข้อความ Error ในหน้า System Logs มาวางที่นี่ได้เลย หรือหากทราบว่าตัว pfSense ของคุณเป็น ฝั่ง Server (รับเชื่อมต่อ) หรือ ฝั่ง Client (วิ่งไปเกาะคนอื่น) จะช่วยให้จำกัดวงแก้ปัญหาได้แคบลง
